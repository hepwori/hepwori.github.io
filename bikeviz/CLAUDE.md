# cycling dashboard

a personal analytics tool for visualising road cycling data extracted from apple health.

## what this is

isaac rides the same routes repeatedly and captures everything via apple workouts. the goal is to surface time-series trends per route — speed, duration, HR, elevation — to track fitness over time. the dataset is years of rides.

the pipeline is two-stage:
1. **parse_health.py** — streams `export.xml` (2–3 GB) using `iterparse`, extracts cycling workouts into `rides.json`, and processes GPX files from `workout-routes/` into `tracks.json`
2. **index.html** — standalone single-file web app served via `python3 -m http.server`; fetches `rides.json` and optionally `tracks.json` automatically on load

no framework. no build step. everything runs locally.

## data model

### rides.json
```json
{
  "generated": "2026-03-22T...",
  "source_file": "export.xml",
  "total_rides": 847,
  "rides": [
    {
      "date": "2021-06-04T07:32:11+00:00",
      "dist_mi": 18.34,
      "dur_mins": 62.4,
      "speed_mph": 17.63,
      "hr_avg": 148.2,
      "hr_max": 171.0,
      "hr_min": 112.0,
      "energy_kcal": 820.5,
      "elev_gain_ft": 1240,
      "source": "Isaac's Apple Watch"
    }
  ]
}
```

all distances in miles. durations in minutes. HR in bpm. elevation in feet. dates are ISO 8601 strings. `elev_gain_ft` is null if no matching GPX file was found.

### tracks.json
```json
{
  "generated": "2026-03-22T...",
  "tracks": {
    "2021-06-04T07:32:11+00:00": [[39.35279, -120.21159, 1926.8], ...]
  }
}
```

keyed by the exact date string from `rides.json`. each track is a list of `[lat, lon, ele_m]` triplets, downsampled to ≤200 points. elevation is in metres (raw from GPX). the dashboard joins on this key using `ride.dateStr`.

## parse_health.py — how it works

1. `iterparse` streams `export.xml` — never loads the whole file into memory
2. prints progress every 5 seconds (workouts scanned, cycling rides found)
3. deduplicates rides with identical `(date, source)` — catches Strava re-sync duplicates
4. auto-detects `workout-routes/` sibling directory and calls `match_gpx_to_rides()`
5. `parse_gpx()` extracts `(lat, lon, ele_m)` tuples from each GPX file, computes elevation gain (sum of positive barometric diffs), downsamples to 200 points
6. matches GPX files to rides by start time (first `<trkpt><time>`, within 2-minute window) — note: `<metadata><time>` is the export date, not the workout time
7. writes `rides.json` (indented) and `tracks.json` (compact, can be ~2–5 MB)

## route clustering

apple health has no concept of named routes. we infer routes by bucketing rides by distance within an epsilon (default ±0.10 mi). rides with 3+ siblings in a bucket become a named route (A, B, C...). the dashboard exposes a live epsilon slider so the user can tune clustering interactively. routes can be renamed in the UI (session-only, not persisted).

clustering uses `allRides` (all loaded rides) regardless of active date filter, so route assignments are stable when filtering. the filter only affects what's displayed.

this is a pragmatic approximation — it works well because isaac does very few distinct routes. it will fail to distinguish routes of similar length on different courses; the route map makes this visible in practice.

## current dashboard features

- **date filter** — year toggle pills (multi-select) + last-N presets (20/50/100/200); filters all charts and metrics. `filteredRides()` is the single source of truth.
- **overview metrics** — total rides, distance, time, routes, total climbing (conditional on elevation data)
- **rides per month** — bar chart
- **distance scatter** — all rides over time
- **cumulative distance** — running total
- **cumulative elevation** — running total (conditional on elevation data)
- **route map** — SVG overlay of all filtered rides on the selected route; segments colored by elevation (blue→green→red, normalised per-route); rendered as 32 color-bucketed paths for performance
- **per-route ride traces** — multi-series chart (x=distance mi, y=elapsed min), one line per ride, colored grey→green by date
- **HR trend** — per-route heart rate over time (conditional on 3+ rides with HR data)

## dashboard architecture notes

- `allRides` — all rides with `.route` assigned; never filtered
- `filteredRides()` — applies `selectedYears` + `lastN` to `allRides`; used everywhere for rendering
- `trackData` — `{dateStr: [[lat,lon,ele_m,elapsed_min],…]}` loaded from tracks.json; joined via `ride.dateStr`
- `assignRoutes()` — always runs on `allRides`; called on load and when epsilon changes
- `refilter()` — rebuilds filter pills + re-renders all charts; called on any filter change
- fetch-based boot: `Promise.allSettled([fetch('./rides.json'), fetch('./tracks.json')])`; tracks.json failure is non-fatal

## known limitations / good next tasks

- **route names don't persist** — renaming is session-only. could use localStorage or a `routes.json` sidecar
- **clustering is distance-only** — two routes of similar length are indistinguishable. real fix requires GPS fingerprinting (start point + shape matching)
- **no segment analysis** — strava-style segment detection would require correlating GPX traces
- **no power data** — isaac doesn't have a power meter; schema supports it if added
- **single-file HTML** — deliberate constraint for portability. if complexity grows, consider a simple vite app
- **CDN dependency** — chart.js loaded from CDN; requires internet on first open. could be bundled
- **HR data sparsity** — HR chart only renders if 3+ rides on the route have HR data

## philosophy

- keep it local — no cloud, no accounts, no telemetry
- keep it simple — one python file, one html file, two json files
- keep it fast — no server, no build step; json files are the only thing the browser loads
- prefer working over clever — distance bucketing is good enough for a small route set

## running it

```bash
# step 1 — generate rides.json + tracks.json (once, or when you re-export)
python3 parse_health.py /path/to/export.xml

# step 2 — serve and open the dashboard
cd /path/to/bikeviz
python3 -m http.server 8000
# open http://localhost:8000
```

python 3.8+ required. no dependencies beyond stdlib. the dashboard needs internet access to load chart.js from CDN.
